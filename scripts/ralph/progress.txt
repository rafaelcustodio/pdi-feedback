## Codebase Patterns
- Server actions use `"use server"` directive and are placed in `actions.ts` next to the page that uses them
- Server actions always check `auth()` and role before executing - return `{ success, error? }` pattern
- Use `revalidatePath()` after mutations to refresh server component data
- Tree structures: use `buildTree()` utility to convert flat Prisma results into nested nodes for rendering
- Employee pages use route pattern: `/colaboradores` (list), `/colaboradores/novo` (create), `/colaboradores/[id]` (edit), `/colaboradores/[id]/hierarquia` (tree view)
- Soft delete pattern: set `isActive = false` on User, close `EmployeeHierarchy.endDate`, preserve all history
- Manager filtering: use `getManagerCandidates(orgUnitId, excludeUserId)` to get managers for a given org unit + admins
- Prisma `findUnique` in while loops can cause implicit `any` - use explicit type casts for included relations
- Prisma v7 requires driver adapters (e.g., `PrismaPg`) in PrismaClient constructor - no more `url` in schema.prisma datasource
- Prisma model `PDI` accessed as `prisma.pDI` (camelCase), `PDIGoal` as `prisma.pDIGoal`, etc.
- Prisma model `Feedback` accessed as `prisma.feedback` (lowercase) - unlike `PDI` which is `prisma.pDI`
- Feedback access: employees see own submitted feedbacks, managers see subordinates, admins see all
- Prisma WHERE composition: use `AND: [baseFilter, ...conditions]` array when combining multiple `OR` clauses - spreading `OR` overwrites previous `OR` keys
- Dual-button save pattern: form submit = save draft, separate button = submit with full validation
- PDI goals managed in transactions: delete removed, update existing, create new - use `prisma.$transaction` for atomicity
- PDI route pattern: `/pdis` (list), `/pdis/novo` (create), `/pdis/[id]` (detail/edit with tracking) - same as feedback pattern
- PDIEvidence accessed as `prisma.pDIEvidence`, PDIComment as `prisma.pDIComment`
- Nested resource access control: fetch parent chain (e.g., evidence → goal → PDI) to verify employee/manager access
- Schedule recalculation: use `recalculatePDISchedule`/`recalculateFeedbackSchedule` from `src/lib/schedule-utils.ts` after completion/submission
- PDISchedule/FeedbackSchedule: `prisma.pDISchedule` and `prisma.feedbackSchedule` - one active schedule per employee (isActive=true)
- Competencies for PDI goals: predefined list in pdi-form.tsx (Liderança, Comunicação, etc.) - stored as string in DB
- Notification model accessed as `prisma.notification` (lowercase) - similar to `prisma.feedback`
- Notification deduplication: embed messageKey `[type_id_date]` in message, check `{ contains: messageKey }` before creating
- Email service: Nodemailer with SMTP config via env vars (SMTP_HOST/PORT/USER/PASS/FROM), fallback to JSON transport (console logging) in dev
- Cron API routes: use `/api/cron/*` pattern, protect with CRON_SECRET Bearer token, exclude from NextAuth middleware matcher
- FeedbackStatus enum: `draft`, `scheduled`, `submitted` - scheduled feedbacks are editable (like drafts), not visible to employees
- NotificationType enum: `pdi_reminder`, `feedback_reminder`, `feedback_scheduled`, `feedback_submitted_auto`, `general`
- UI status label maps: update `statusLabels` Record and color classes in feedback-table.tsx and feedback-read-only.tsx when adding new statuses
- Migration files created manually in `prisma/migrations/YYYYMMDDHHMMSS_description/migration.sql` (no DB connection needed)
- Cron routes: `/api/cron/email-reminders` (schedule notifications + emails), `/api/cron/scheduled-feedbacks` (auto-submit scheduled feedbacks)
- Per-item try/catch in cron loops: allows partial success, failed items don't block others, failure notifications created for managers
- Middleware exclusions: add new API patterns to the regex negative lookahead in `src/middleware.ts`
- Draft feedback with scheduledAt = "Agendado para preenchimento" (planned fill-in date); scheduled feedback with scheduledAt = "Agendado" (auto-submit date)
- Avoid `setState` in `useEffect` for prop syncing - ESLint `react-hooks/set-state-in-effect` rule - use `useMemo` with derived state instead
- PDIGoal.dueDate is nullable (`DateTime?`) - always guard with null check or `{ not: null }` in Prisma where
- Prisma v7 generated client output is at `src/generated/prisma/client` - import from `@/generated/prisma/client`
- Prisma binary engines (binaries.prisma.sh) blocked by firewall - use `PRISMA_SCHEMA_ENGINE_BINARY=/tmp/fake-schema-engine` workaround for CLI commands (validate, generate)
- npm install requires `--no-bin-links` and `--ignore-scripts` flags due to 9p filesystem (WSL2 mount) - then install prisma separately with scripts enabled
- TypeScript: `node node_modules/typescript/bin/tsc --noEmit` (no npx bin links)
- ESLint: `node node_modules/eslint/bin/eslint.js`
- Prisma CLI: `PRISMA_SCHEMA_ENGINE_BINARY=/tmp/fake-schema-engine node node_modules/prisma/build/index.js <command>`
- Project uses Next.js 16 App Router + Tailwind CSS v4 + lucide-react for icons
- Portuguese (pt-BR) language for UI labels and content
- Layout uses `(dashboard)` route group with `AppLayout` wrapper component
- Sidebar nav items defined in `src/components/layout/sidebar.tsx` - add new routes there
- Use `next/image` instead of `<img>` to avoid ESLint warnings
- NextAuth v5 (next-auth@beta 5.0.0-beta.30) with `AUTH_SECRET` env var (also supports `NEXTAUTH_SECRET`)
- MicrosoftEntraID provider uses `issuer` URL not `tenantId` - format: `https://login.microsoftonline.com/{tenantId}/v2.0`
- SSO provider string in NextAuth callbacks is `"microsoft-entra-id"` (the provider id)
- Auth config at `src/lib/auth.ts` - exports `handlers`, `signIn`, `signOut`, `auth`
