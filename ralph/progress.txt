## Codebase Patterns
- Prisma schema at `prisma/schema.prisma`, generated client at `src/generated/prisma/` (gitignored)
- Server Actions pattern: `src/app/(dashboard)/[feature]/actions.ts` for mutations/queries
- Access control: always use `getEffectiveAuth()` from `@/lib/impersonation` (never `auth()` directly)
- Employee form: `src/components/employee-form.tsx` ‚Äî main form for creating/editing employees
- Profile page: `src/app/(dashboard)/perfil/` ‚Äî employee self-service view
- Email: `src/lib/email.ts` + `src/lib/email-templates.ts` ‚Äî Nodemailer with SMTP fallback to console
- Notifications: existing `Notification` model in Prisma schema
- Dark mode: always include `dark:` Tailwind variants
- Timezone: use UTC methods for dates from server (getUTCDate, etc.)
- Prisma generate workaround: `PRISMA_SCHEMA_ENGINE_BINARY=/tmp/schema-engine npx prisma generate` (create mock schema-engine first: `echo '#!/bin/sh' > /tmp/schema-engine && chmod +x /tmp/schema-engine`)
- Migration files must be created manually (no network access to binaries.prisma.sh for schema-engine)
- Access control for profile read: use `canAccessEmployee()` from `@/lib/access-control` ‚Äî admin=all, manager=subordinates, employee=self
- Role-based field filtering: `isRestricted = role === "manager" && !isOwnProfile` ‚Äî restricted users see null for sensitive fields
- updateEmployeeDependents/EmergencyContacts pattern: diff incoming vs existing IDs ‚Üí delete removed, update existing, create new
- Pre-existing lint errors (7 errors, 8 warnings) and 8 test failures in sector-schedule-utils.test.ts ‚Äî not caused by our changes
- Enum fields in Prisma create/update: use `(value as never) || null` to satisfy type-checker for enum columns
- Boolean fields in Prisma: use `data.field ?? null` (not `|| null`) since `false` is a valid value
- Employee form tabs: uses simple `activeTab` state (0-4), all form state at top level so switching tabs preserves data
- Dark mode in forms: use shared CSS class strings (`inputClass`, `labelClass`, `sectionClass`) for consistency
- Enum select options: defined as const arrays at module scope (e.g., `GENDER_OPTIONS`, `ETHNICITY_OPTIONS`)
- Dynamic list pattern (dependents/contacts): use array state + map with index for editing, filter on idx !== i for removal, spread + new item for addition
- Form types for dynamic lists: define separate types (DependentFormItem, EmergencyContactFormItem) with optional `id` for new vs existing items
- getEmployeeById includes `dependents` and `emergencyContacts` with select/orderBy for the edit page
- Prisma JSON fields: use `?? undefined` (not `?? null`) for socialNetworks in create/update to avoid type error with NullableJsonNullValueInput
- Multi-select checkboxes pattern: array state + includes check + filter/spread for toggle
- Social networks stored as JSON object `{networkName: handle}` ‚Äî use `Object.keys()` to derive selected networks
- Composite string values: hasPets with "outra" uses `"outra: description"` format ‚Äî parse with startsWith on init
- ChangeRequest values are stored as strings ‚Äî use `parseChangeRequestValue(fieldName, value)` helper to convert to boolean/Date/string before applying to User model
- ChangeRequest duplicate prevention: check for existing pending request on same userId+fieldName before creating new one
- Notification to all admins pattern: `prisma.user.findMany({ where: { role: "admin", isActive: true } })` + `prisma.notification.createMany` + fire-and-forget emails with `.catch()`
- Adding new NotificationType enum value: create migration with `ALTER TYPE "NotificationType" ADD VALUE 'value_name';`
- Profile editing pattern: tabs 1-4 use ChangeRequest flow (createChangeRequest per changed field), tab 5 saves directly via updateEmployeeProfile
- ChangeRequest field comparison: serialize profile values to strings for old/new values ‚Äî booleans as "true"/"false", dates as ISO date string, enums as raw value
- Pending CR indicators: use `Set<string>` of field names from pendingChangeRequests, show amber PendingBadge component, disable inputs for pending fields
- Profile edit flow: `getMyPendingChangeRequests` server action fetches pending CRs, passed from page ‚Üí tabs ‚Üí perfil-data via props, router.refresh() after save
- Colaboradores page tabs: `colaboradores-tabs.tsx` ‚Äî uses URL search params per tab (search/page, psearch/ppage, crsearch/crpage/crstatus) to preserve state across navigation
- Change requests table: `change-requests-table.tsx` ‚Äî standalone table component with FIELD_LABELS/ENUM_LABELS maps for Portuguese localization of field names and enum values
