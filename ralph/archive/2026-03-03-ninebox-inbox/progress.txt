## Codebase Patterns
- Prisma schema at `prisma/schema.prisma`, generated client at `src/generated/prisma/` (gitignored)
- In this sandboxed dev environment, `prisma generate` and `prisma migrate` cannot download engine binaries (network restricted). Manually create migration SQL and update generated client files (enums.ts, models.ts, model type files) instead.
- Migration SQL files go in `prisma/migrations/YYYYMMDDHHMMSS_name/migration.sql`
- Generated enums live in `src/generated/prisma/enums.ts`, model types in `src/generated/prisma/models/`
- Existing lint errors in theme-provider.tsx and programacao/actions.ts are pre-existing; lint will always show 1 error + 4 warnings
- Server Actions pattern: `src/app/(dashboard)/[feature]/actions.ts` for mutations/queries
- Access control: always use `getAccessibleEmployeeIds()` or filter helpers before querying
- To use NineBox models in Prisma queries, add `nineBoxEvaluation: any; nineBoxEvaluator: any` to `PrismaClientExtended` in `src/lib/prisma.ts` and add delegate getters to `src/generated/prisma/internal/class.ts`
- Modals follow pattern: `useState` boolean + `fixed inset-0 z-50 bg-black/50` overlay + centered white card
- NineBox server actions live in `src/app/(dashboard)/feedbacks/ninebox-actions.ts` (separate from main actions.ts) for feedback-related operations
- NineBox dashboard server actions live in `src/app/(dashboard)/colaboradores/ninebox-actions.ts` for dashboard/reporting data
- NineBoxGrid is a reusable pure component at `src/components/ninebox-grid.tsx` — takes desempenhoFaixa and potencialFaixa props
- NineBoxDashboard is a server component at `src/components/ninebox-dashboard.tsx` — takes current and previous NineBoxEvalResult
- Employee-facing NineBox view uses `src/app/(dashboard)/perfil/actions.ts` for `getMyNineBoxResult` — only returns closed evaluations, strips evaluator details
- Perfil page uses tab pattern: `PerfilTabs` client component with `useState` for tab switching, server component passes pre-fetched data as ReactNode
- Routes outside the (dashboard) group (e.g., `/ninebox/[evaluatorId]`) are still auth-protected by middleware matcher; they just don't have the dashboard sidebar/layout
- For standalone form pages outside the dashboard, create route at `src/app/[feature]/[param]/page.tsx` as a server component that calls `auth()` and renders a client form component
- Dashboard page at `src/app/(dashboard)/dashboard/page.tsx` uses `getDashboardData()` from `./actions.ts`; `DashboardData` type exported from actions; add new queries alongside existing ones
- `getMyNineBoxEvaluations()` returns `MyNineBoxEvaluations` type with `pending` and `completed` arrays — types exported from dashboard actions.ts
